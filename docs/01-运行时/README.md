## 30 秒结论

- 当前主运行单元是 `iam-apiserver`，HTTP 与 gRPC 都由同一个进程承载。
- 运行时层现在主要回答 3 件事：
  - 进程怎么启动，HTTP / REST 路由和模块怎么装配
  - gRPC 服务怎么注册，mTLS / ACL / 健康检查怎么挂上去
  - HTTP JWT 中间件今天到底保护了哪些路由，又把什么身份信息塞进了上下文
- 另外还有一类经常被误讲的运行面问题：健康检查、debug 路由和“部分初始化也继续启动”的边界，这一组也已经单独补文。
- 契约字段和 RPC 名称不是这一层的重点；那部分应回到 `api/` 和 [../03-接口与集成](../03-接口与集成/README.md)。

## 1. `01-运行时` 这一组文档今天到底负责什么

运行时层只回答“系统今天怎么跑”：

- 进程入口与装配链
- HTTP / gRPC 暴露面
- JWT 中间件、mTLS、ACL、健康检查这些运行面能力
- 降级启动、debug 路由、探针这些容易被误讲的运行时边界

它不负责重复维护业务对象、接口字段或集成契约细节。

## 3. 这一组文档里哪些运行时术语值得统一

先统一这几个词：

- `主运行单元`：今天指 `iam-apiserver`，不是多进程集群视角
- `装配`：指 `container / assembler` 把模块 wiring 进进程
- `暴露面`：指已经注册进当前 HTTP / gRPC 服务器的接口面
- `条件式保护`：指依赖某模块存在时才挂中间件或能力
- `降级启动`：指依赖或模块不完整时，进程仍继续启动

## 4. 这一组文档的图表节奏应该怎么把握

- 入口页不堆图，只先说明这组文档回答什么问题。
- 服务入口文优先用启动链图和装配图。
- gRPC 文优先用运行时角色图和安全链图。
- JWT 中间件文优先用请求链图。
- 健康检查文优先用探针分层图和降级启动流程图。

换句话说，图只在第一次解释那个运行时问题时出现，不做装饰性重复。

## 6. 当前边界

- 这一组回答“系统今天怎么跑”，不回答“字段长什么样”；字段和 RPC 以 `api/` 契约为准。
- 这一组回答“哪些路由和服务被注册进进程”，不重复解释 `authn / authz / 用户域（uc）` 的业务模型。
- 这一组会把“当前已保护 / 当前未保护 / 条件式降级（如认证模块未起时 no-op）”写清，不把运行时设计意图包装成已落地能力。
