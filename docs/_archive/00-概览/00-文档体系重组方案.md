# iam-contracts 文档体系重组方案

本文回答：`iam-contracts` 的文档体系为什么不应继续以“纯模块拆分”为顶层结构，新的顶层目录如何设计，以及当前迁移已经推进到哪一步。

## 30 秒结论

- `iam-contracts` 的顶层文档结构更适合采用“**系统维度优先，业务域为其中一层**”的方案，而不是继续只按 `认证域 / 授权域 / 用户域` 拆。
- 顶层建议重组为：`00-概览`、`01-运行时`、`02-业务域`、`03-接口与集成`、`04-基础设施与运维`、`05-专题分析`。
- `api/` 继续独立保留为**机器契约层**，不并入 `docs/`，但需要在 `docs/` 根入口里被明确标成事实来源。
- 当前新框架已经覆盖 `00-概览 / 01-运行时 / 02-业务域 / 03-接口与集成 / 04-基础设施与运维 / 05-专题分析`；遗留工作主要是继续拆分混合文档，而不是再扩顶层目录。
- 当前迁移已完成业务域主轴建设：`docs/02-业务域/` 已建立，`authn`、`authz`、`user` 新正文已经迁入；旧 `01-认证域/`、`02-授权域/`、`03-用户域/` 已归档到 [../_archive/README.md](../_archive/README.md)。
- 后续仍建议采用“先建新骨架，再逐篇迁移，最后清旧目录”的方式，不建议一次性硬改所有目录名。

## 为什么要改

当前结构的优点是业务边界清楚，`认证 / 授权 / 用户` 这三组文档适合讲 DDD 和领域边界。但它也有三个明显限制：

1. **系统问题没有自然归位**  
   运行时、gRPC/mTLS、部署、配置、契约、Makefile 这些内容并不属于任何单一业务域，只能被放在边上。

2. **读者路径被迫走“模块视角”**  
   新读者、集成方、运维、接口消费者常常先关心“系统怎么跑”“我应该看哪个合同”“端口和证书怎么配”，而不是先关心某个领域模型。

3. **混合文档越来越多**  
   例如当前的 `gRPC 服务设计` 同时混合了运行时、安全、证书、合同使用方式；这类内容继续放在“概览”下面会越来越难维护。

因此，`业务域` 仍然应该保留，但更适合作为顶层结构中的**一个层**，而不是整个文档体系的唯一主轴。

## 目标顶层结构

### `docs/00-概览`

负责建立系统级心智模型，回答“这是什么系统、代码如何组织、事实来源在哪里、应该从哪里开始读”。

适合放：

- 系统地图 / 系统架构总览
- 核心概念术语
- 代码组织与边界
- 文档体系说明

### `docs/01-运行时`

负责讲清运行时形态和跨层调用路径，回答“这个系统怎么启动、怎么对外暴露、运行面怎么组织”。

适合放：

- `iam-apiserver` 进程说明
- 启动与装配
- REST / gRPC 暴露面
- gRPC 与 mTLS
- 认证与身份链路
- 配置装载与运行时依赖

### `docs/02-业务域`

负责讲清业务边界，是新的“模块层”。

当前做法不是继续复制旧目录，而是直接在这一层写“现状版正文”。已经迁入的正文有：

- [01-authn-认证、Token、JWKS.md](../02-业务域/01-authn-认证、Token、JWKS.md)
- [02-authz-角色、策略、资源、Assignment.md](../02-业务域/02-authz-角色、策略、资源、Assignment.md)

当前做法不是继续复制旧目录，而是直接在这一层写“现状版正文”。已经迁入的正文有：

- [03-user-用户、儿童、Guardianship.md](../02-业务域/03-user-用户、儿童、Guardianship.md)

这里继续保留 DDD 和领域模型文档，但顶层不再让它们独占整个信息架构。

### `docs/03-接口与集成`

负责讲“对外合同和集成方式”，回答“我怎么接、我应该看哪份合同、接入校验怎么做”。

适合放：

- REST 契约说明
- gRPC 契约说明
- QS 接入 IAM 指南
- 错误语义、认证方式、分页约定
- 外部系统接入说明

说明：

- `api/` 依然是机器契约层
- `docs/03-接口与集成` 是对契约层的解释层

### `docs/04-基础设施与运维`

负责讲横切技术机制和交付/运行入口，回答“技术底座是什么、怎么部署、怎么验、怎么迁移”。

适合放：

- 六边形架构
- CQRS
- MySQL / Redis / Casbin 等基础设施集成方式
- Makefile 指南
- Docker / Compose / systemd
- 端口、迁移、配置与运维入口

### `docs/05-专题分析`

负责承载跨越多个层次、多个域的深度主题，回答“某条主链路或某个关键设计到底是如何成立的”。

适合放：

- 登录与令牌链路
- 权限判定链路
- 监护关系链路
- gRPC 证书与密钥治理
- 契约校验链
- Suggest / IDP 这类专项设计专题

## `api/` 的位置

新的顶层方案**不改变** `api/` 的位置：

- `api/rest/*.yaml` 继续作为 REST 机器契约层
- `api/grpc/**/*.proto` 继续作为 gRPC 机器契约层
- `docs/` 只负责解释这些合同如何组织、如何消费、如何验证

换句话说，新的 `docs/` 是“系统解释层”，不是要把 `api/` 吞进去。

## 需要拆分而不是平移的文档

这类文档如果整篇搬家，会把旧结构的问题一起搬过去：

### 1. `gRPC 服务设计`（已完成拆分）

当前同时混合了：

- 运行时服务间通信
- mTLS 与证书治理
- 契约消费方式
- 项目间接入说明

现已拆成两篇：

- `01-运行时/02-gRPC与mTLS.md`
- `03-接口与集成/02-gRPC契约与接入.md`
- 如内容仍多，再追加 `05-专题分析/证书与密钥治理.md`

### 2. `ops/README.md`

当前已经承担过运维入口职责，但现行入口已经迁到 `docs/04-基础设施与运维/`，旧目录应只保留为历史资料。

### 3. `suggest.md`（已完成迁移）

这篇旧稿原先挂在根目录，既不像系统总览，也不像运维文档。现已按“补充业务能力正文”迁入：

- `docs/02-业务域/04-suggest-儿童联想搜索.md`

原稿保留在：

- `_archive/suggest.md`

## 迁移原则

- **先建新骨架，再迁正文**：先把目标目录和入口页写出来，再逐篇迁移内容。
- **不一次性硬改所有文件名**：大规模 rename 容易让链接、引用、读者入口同时失效。
- **混合文档优先拆分**：优先处理 `gRPC 服务设计`、`ops/` 这类跨层文档。
- **业务域文档保持成组迁移**：`authn`、`authz`、`uc` 这三组不要拆散到不同阶段，否则边界会再次模糊。
- **契约层独立**：`api/` 不并入 `docs/`，只在新的入口层中被明确标成事实来源。

## 验收标准

- 新读者能在 30 秒内看懂 `docs/` 的层次和入口。
- 读者能区分：系统解释层、业务域层、接口契约层、基础设施/运维层。
- 运行时、接口、运维不再被迫挂在“模块之外的边角位置”。
- `authn / authz / uc` 仍然保留为稳定的业务域层，不因重组而失去边界清晰度。
- `api/` 继续保持机器契约层地位，没有被 prose 文档替代。
