# 核心优先迁移清单

本文回答：`iam-contracts` 在新文档框架已经建立的前提下，哪些内容属于必须优先深挖的核心，哪些属于支撑层，哪些可以后置维护，以及下一阶段最值得投入的迁移顺序是什么。

## 30 秒结论

- `iam-contracts` 的核心不是“目录有多少层”，而是四件事是否讲透：**身份对象模型、认证链、授权判定链、对外契约**。
- 因此文档维护不应平均用力，而应优先深挖 `认证域 / 授权域 / 用户域 / 接口与集成`，再补运行时和基础设施。
- 新框架已经基本立住：`总览 / 运行时 / 业务域 / 接口与集成 / 基础设施与运维`。接下来更重要的是按核心度补正文，而不是继续平均重构所有入口页。
- REST / gRPC 契约解释层、授权接入边界、身份接入边界、认证链专题、授权判定链专题和监护关系链专题都已经建立；当前更值得继续深挖的是 **混合文档拆分与核心主线细节对齐**。

## 1. 先定义什么是“核心”

对 `iam` 来说，最值得优先维护的不是某个目录，而是这四个问题：

1. 系统到底管理哪些身份对象  
2. 调用方如何完成认证  
3. 系统如何完成授权判定  
4. 外部系统到底该怎么接入这些能力  

如果这四件事没有讲透，文档即使很全，也很难称得上“抓住核心”。

接口与集成层最关键的 4 篇解释文档已经建立。下一步如果继续按“缺什么最影响核心表达”来排，更值得投入的是：

相比继续新增更多入口页，这两件事更能收紧“用户域事实层”和“集成边界层”之间的口径。

## 6. 建议迁移顺序

### Phase A：先把核心入口站稳

- 保持 [01-系统架构总览.md](./01-系统架构总览.md) 为单一总入口
- 保持 [../02-业务域/README.md](../02-业务域/README.md) 为业务主轴入口
- 保持 [../03-接口与集成/README.md](../03-接口与集成/README.md) 为契约解释层入口

### Phase B：优先深挖 `P0`

先按这个顺序：

1. 认证域
2. 授权域
3. 用户域
4. REST / gRPC 契约解释层

### Phase C：补专题主线

补 3 篇核心专题：

1. 认证链
2. 授权判定链
3. 监护关系链

### Phase D：再补 `P1`

- 运行时
- 基础设施
- 运维

### Phase E：最后再收 `P2`

- Suggest
- Collection 集成
- 细碎运维与非核心扩展

## 7. 验收标准

如果这一套“核心优先”策略执行得好，最后应满足：

- 新读者能在 10 分钟内理解这个 IAM 的主轴
- 集成方能快速找到 REST / gRPC 契约解释层
- 维护者不会把大量精力先花在非核心支撑文档上
- 业务域文档和接口文档成为优先维护对象，而不是运维细节先于主轴不断膨胀

## 8. 当前建议

如果下一步只做最值当的一件事，我建议直接开始补：

1. 对齐 `02-业务域` 下的 `user` 正文与当前已建立的接口边界文档
2. 然后再决定是否补更细的接入专题

这比继续平均打磨更多支撑层文档更符合“核心优先”的策略。
