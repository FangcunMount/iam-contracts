# 专题分析

## 本文回答

本组 README 只回答 5 件事：

1. 为什么 `05-专题分析` 要单独存在
2. 当前这 7 篇专题分别讲什么，应该先看哪篇
3. 它和 `02-业务域`、`03-接口与集成`、`01-运行时` 怎么分工
4. 新增缓存专题与 SDK 专题后，这一组的阅读顺序怎么选
5. 这组文档后续还适合继续扩什么问题

## 30 秒结论

> **一句话**：`05-专题分析` 负责把 `iam-contracts` 里“跨层才能讲清楚”的主链路和设计判断单独拉出来讲。它不替代业务域正文，也不替代契约层，而是把 **对象、运行时链路、工程边界、当前保证与风险边界** 串成读者可以一口气读懂的专题文。

| 问题 | 当前答案 |
| ---- | ---- |
| 为什么需要这一组 | 因为认证、授权、监护关系、缓存层、Redis 建模判断、SDK 接入价值都不是单一模块能讲清的 |
| 当前已有哪些专题 | 认证链、认证语义拆层、授权判定链、监护关系链、缓存层设计与治理、缓存数据结构判断、SDK 封装与接入价值 |
| 这组最适合做什么 | 讲跨层主链、设计取舍、工程边界、当前风险 |
| 这组不适合做什么 | 不替代 `02-业务域` 写静态模型，也不替代 `api/*` 写契约真值 |

## 为什么需要这一组

如果只看 `02-业务域`，你能知道：

- 模块边界
- 领域对象
- 应用服务

但你不一定能一口气看懂：

- 登录请求是怎么变成 `Session + Access / Refresh Token`
- 授权管理链如何落到 Casbin 判定
- 监护关系如何同时影响写链、读链和访问边界
- Session 为什么会把 IAM Redis 建模从“全是 String”推进到“String + ZSet”
- SDK 为什么既不是新真值层，也不是薄 wrapper

这组文档就是专门回答这些“跨层才能讲清的问题”。

## 后续还适合扩什么

如果继续扩专题，这一组更适合继续新增：

- 在线 verify 与离线 JWKS 的接入决策专题
- Session 管理与管理员控制面的专题
- IAM 与业务系统竖切边界专题

原则不变：

- 只有“跨层才能讲清”的问题，才值得进 `05-专题分析`
- 静态模型回 `02-业务域`
- 契约与接入方式回 `03-接口与集成`
