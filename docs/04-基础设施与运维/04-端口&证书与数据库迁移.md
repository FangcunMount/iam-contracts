# 端口、证书与数据库迁移

本文回答：`iam-contracts` 当前 dev/prod 端口如何分配，gRPC 证书与 ACL 从哪里看，Docker 工件怎样组织，以及数据库迁移今天应该以哪些文件和目录为准。

## 30 秒结论

- 端口真值以 [../../configs/apiserver.dev.yaml](../../configs/apiserver.dev.yaml) 和 [../../configs/apiserver.prod.yaml](../../configs/apiserver.prod.yaml) 为准：dev 是 `18081 / 18441 / 19091 / 19092`，prod 是 `9080 / 9444 / 9090 / 9091`。
- gRPC 安全边界主要由 `grpc.mtls` 和 [../../configs/grpc_acl.yaml](../../configs/grpc_acl.yaml) 共同定义；证书说明优先看 [../../scripts/cert/README.md](../../scripts/cert/README.md)。
- Docker 交付入口以 [../../build/docker/README.md](../../build/docker/README.md) 与 `build/docker/*` 为准；`Makefile` 只是这些工件的统一调用面。
- 数据库迁移真值在 [../../internal/pkg/migration/](../../internal/pkg/migration/)；版本由 `internal/pkg/migration/migrations/*.sql` 文件名手工递增管理，当前已存在 `000001` 到 `000003`。
- `db-init / db-migrate / db-reset` 已被标成弃用；现行更可信的数据库入口是 migration 目录本身和 `make db-bootstrap / db-status / db-backup`。

## 1. 当前端口矩阵

| 面 | dev | prod | 真实配置 |
| ---- | ---- | ---- | ---- |
| HTTP REST | `18081` | `9080` | `insecure.bind-port` |
| HTTPS REST | `18441` | `9444` | `secure.bind-port` |
| gRPC | `19091` | `9090` | `grpc.bind-port` |
| gRPC 健康检查 | `19092` | `9091` | `grpc.healthz-port` |

这些值直接来自：

- [../../configs/apiserver.dev.yaml](../../configs/apiserver.dev.yaml)
- [../../configs/apiserver.prod.yaml](../../configs/apiserver.prod.yaml)

因此端口相关的 prose 文档如果和这两个配置文件冲突，以配置文件为准。

## 2. gRPC 证书与 ACL

### 2.1 mTLS 真值位置

当前 gRPC 的证书与 mTLS 主要看两处：

1. `configs/apiserver.dev.yaml` / `configs/apiserver.prod.yaml` 里的 `grpc.mtls`
2. [../../configs/grpc_acl.yaml](../../configs/grpc_acl.yaml)

其中配置文件里可以直接看到：

- `enabled`
- `require-client-cert`
- `ca-file`
- `server-cert-file`
- `server-key-file`
- `allowed-ous`

### 2.2 当前更可信的证书说明

当前更可信的证书说明入口是：

- [../../scripts/cert/README.md](../../scripts/cert/README.md)
- `make grpc-cert-verify`
- `make grpc-cert-info`

不能直接包装成稳定入口的是：

- `make cert-gen`
- `make cert-test`
- `make test-dev-config`

因为这些目标仍引用缺失脚本。

## 3. Docker 与部署入口

### 3.1 当前工件

Docker 相关工件主要在：

- [../../build/docker/Dockerfile](../../build/docker/Dockerfile)
- [../../build/docker/docker-compose.dev.yml](../../build/docker/docker-compose.dev.yml)
- [../../build/docker/docker-compose.prod.yml](../../build/docker/docker-compose.prod.yml)
- [../../build/docker/README.md](../../build/docker/README.md)

### 3.2 当前理解方式

更准确的口径是：

- Docker / Compose 的真实行为以 `build/docker/*` 为准
- `make docker-*`、`make docker-compose-*`、`make docker-dev-*` 是这些工件的统一入口
- 端口和网络说明仍应回到 compose 文件和配置文件本身核对

## 4. 数据库迁移

### 4.1 当前 migration 真值

当前 migration 相关真值不在旧运维手册，而在：

- [../../internal/pkg/migration/README.md](../../internal/pkg/migration/README.md)
- [../../internal/pkg/migration/MIGRATION_GUIDE.md](../../internal/pkg/migration/MIGRATION_GUIDE.md)
- [../../internal/pkg/migration/migrate.go](../../internal/pkg/migration/migrate.go)
- [../../internal/pkg/migration/migrations/](../../internal/pkg/migration/migrations/)

### 4.2 当前版本状态

当前仓库已经存在：

- `000001_init_schema`
- `000002_add_guardianships_child_id_index`
- `000003_make_users_phone_nullable`

这说明当前 migration 已经不是旧文档里举例的“000002_seed_data”那套口径，文档必须以实际 SQL 文件为准。

### 4.3 当前该怎么表述

今天最准确的 migration 口径是：

- 版本通过 `migrations/*.sql` 文件名手工管理
- SQL 文件通过 `go:embed` 被打进二进制
- 应用启动时 migration 代码会检查并执行未应用版本
- 若要新增 migration，优先对照 migration 目录和现有文件命名规则，而不是照抄旧运维手册里的示例版本号

### 4.4 当前边界

`Makefile` 里已经明确写了：

- `db-init`
- `db-migrate`
- `db-reset`

这些是**弃用**入口。  
现行更可信的数据库辅助命令是：

- `make db-bootstrap`
- `make db-status`
- `make db-backup`
- `make db-connect`
