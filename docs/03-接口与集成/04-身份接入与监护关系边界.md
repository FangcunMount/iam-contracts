# 身份接入与监护关系边界

本文回答：外部系统今天能怎样接入 `iam-contracts` 的身份与监护关系能力，REST 与 gRPC 各自更适合什么场景，哪些契约已经落地，哪些地方仍存在合同漂移、实现缺口或不应讲过头的边界。

## 30 秒结论

- `已实现`：当前身份与监护能力分成两类暴露面。REST 主要是“当前登录用户上下文”风格的 `/api/v1/identity/*`；gRPC 主要是“服务间显式传 ID”风格的 `IdentityRead`、`GuardianshipQuery`、`GuardianshipCommand`、`IdentityLifecycle`。
- `更适合的接法`：前端或 BFF 若围绕“当前用户是谁、我能看哪些孩子”来接，优先看 REST；后端服务若要显式做 `GetUser / GetChild / IsGuardian / ListGuardians`，gRPC 暴露面当前更完整。
- `当前风险`：`children/register` 不是原子闭环，而是“先建 child，再建 guardianship”的两段事务；OpenAPI 里的 `GET /identity/guardians` 还没接到 router；`revoked_at` 没有在查询判定链里被统一过滤，`active` 过滤当前也是 stub。
- `合同漂移`：REST `children/register` 合同写 `201`，运行时实际走 `200`；REST 的 `user_id / child_id` 查询参数与 binder 的 `userId / childId` 不一致；REST 的 `relation=guardian` 和 gRPC 的 `GRANDPARENT` 在应用层都会落成非预期值。
- `待补证据`：`identity.proto` 里有 `UpdateGuardianRelation` 和 `IdentityStream`，但前者当前返回 `Unimplemented`，后者没有发现运行时注册。

## 1. 先区分两类接法：当前用户上下文 vs 显式 ID 查询

当前身份与监护能力，最好先按调用风格拆成两类：

| 风格 | 当前主要入口 | 更适合什么场景 |
| ---- | ---- | ---- |
| 当前用户上下文 | REST `/api/v1/identity/*` | 前端、BFF、用户自助资料与“我的孩子” |
| 显式 ID 查询/命令 | gRPC `IdentityRead / GuardianshipQuery / GuardianshipCommand` | 服务间集成、显式传 `user_id / child_id` 的查询与判定 |

这意味着今天更准确的接法是：

- “当前是谁、我自己能看什么”这类场景，优先看 REST
- “我想判定某个用户是不是某个孩子的监护人”这类场景，优先看 gRPC

不要一上来就把 REST 和 gRPC 当成完全等价的两套壳。当前它们的覆盖范围并不对称。

## 2. 当前可以怎么接

### 2.1 面向当前登录用户的 REST

当前 router 已注册、且与身份主线最相关的 REST 路径包括：

| 路径 | 当前作用 |
| ---- | ---- |
| `GET /api/v1/identity/me` | 查询当前用户资料 |
| `PATCH /api/v1/identity/me` | 更新当前用户昵称 / 联系方式 |
| `GET /api/v1/identity/me/children` | 查询当前用户监护的儿童 |
| `POST /api/v1/identity/children/register` | 注册儿童并把当前用户设为监护人 |
| `GET /api/v1/identity/children/{id}` | 以“当前用户是监护人”为前提读 child |
| `PATCH /api/v1/identity/children/{id}` | 以“当前用户是监护人”为前提改 child |
| `POST /api/v1/identity/guardians/grant` | 显式授予某个用户对某个孩子的监护关系 |

对应代码在：

- [../../internal/apiserver/interface/uc/restful/router.go](../../internal/apiserver/interface/uc/restful/router.go)
- [../../internal/apiserver/interface/uc/restful/handler/user.go](../../internal/apiserver/interface/uc/restful/handler/user.go)
- [../../internal/apiserver/interface/uc/restful/handler/child.go](../../internal/apiserver/interface/uc/restful/handler/child.go)
- [../../internal/apiserver/interface/uc/restful/handler/guardianship.go](../../internal/apiserver/interface/uc/restful/handler/guardianship.go)

这条 REST 面更像“带用户态语义的业务接口”，而不是“通用身份查询 API”。

### 2.2 面向服务间集成的 gRPC

当前运行时已注册的身份相关 gRPC 服务包括：

| Service | 当前更适合的用途 |
| ---- | ---- |
| `IdentityRead` | `GetUser / GetChild / BatchGet*` 这类显式读请求 |
| `GuardianshipQuery` | `IsGuardian / ListChildren / ListGuardians` |
| `GuardianshipCommand` | `AddGuardian / RevokeGuardian / BatchRevoke / ImportGuardians` |
| `IdentityLifecycle` | 用户创建、更新、状态变更 |

对应代码在：

- [../../internal/apiserver/server.go](../../internal/apiserver/server.go)
- [../../internal/apiserver/interface/uc/grpc/identity/service.go](../../internal/apiserver/interface/uc/grpc/identity/service.go)
- [../../internal/apiserver/interface/uc/grpc/identity/service_impl.go](../../internal/apiserver/interface/uc/grpc/identity/service_impl.go)

如果你的集成场景是：

- 通过 `user_id / child_id` 显式做监护判定
- 列出某个孩子的所有监护人
- 用内部服务直接读 user / child

那么 gRPC 当前通常比 REST 更直接，也更完整。

### 2.3 一个简单的“推荐接法”表

| 场景 | 当前更推荐的入口 | 原因 |
| ---- | ---- | ---- |
| 当前用户查看自己的资料 | REST `/identity/me` | 直接读 JWT 上下文里的当前用户 |
| 当前用户查看自己监护的孩子 | REST `/identity/me/children` | 语义就是“我的孩子” |
| 系统间按 ID 查询用户或儿童 | gRPC `GetUser / GetChild / BatchGet*` | 显式传 ID，比 REST `/me` 风格更合适 |
| 判定 `user_id + child_id` 是否存在监护关系 | gRPC `IsGuardian` | REST 没有稳定暴露的等价判定接口 |
| 列出孩子的监护人 | gRPC `ListGuardians` | REST `/identity/guardians` 还没接到 router |
| 注册儿童并自动绑定当前用户 | REST `POST /identity/children/register` | 当前没有对应 gRPC child-create 接口 |

## 3. 当前不要怎么接

### 3.1 不要默认 `GET /identity/guardians` 已经可用

[../../api/rest/identity.v1.yaml](../../api/rest/identity.v1.yaml) 当前声明了：

- `GET /identity/guardians`
- `POST /identity/guardians/grant`

但 [../../internal/apiserver/interface/uc/restful/router.go](../../internal/apiserver/interface/uc/restful/router.go) 当前只注册了：

- `POST /api/v1/identity/guardians/grant`

`GuardianshipHandler.List()` 虽然存在，也写了 Swagger 注解，但还没有被 router 接起来。  
所以今天不能把“监护关系 REST 列表查询已对外暴露”讲成现状。

### 3.2 不要把 `children/register` 当成原子事务

[../../internal/apiserver/interface/uc/restful/handler/child.go](../../internal/apiserver/interface/uc/restful/handler/child.go) 的 `RegisterChild()` 当前顺序是：

1. `childApp.Register(...)`
2. `guardApp.AddGuardian(...)`
3. 再查询一次 guardianship 组回包

这不是一个“单事务里同时创建 child + guardianship”的闭环，而是两段独立事务。  
所以更准确的口径是：

- 当前支持“注册儿童并自动授监护”
- 但当前不是原子事务

### 3.3 不要默认撤销关系已经在查询链里被统一过滤

当前几条关键链路都没有显式过滤 `revoked_at`：

- repo 的 `FindByUserIDAndChildID`
- repo 的 `FindByUserID`
- repo 的 `FindByChildID`
- repo 的 `IsGuardian`

而 REST `active` 过滤当前也是 stub，`GuardianshipResult` 里也不带 `RevokedAt`。  
这意味着今天不能直接讲成：

- “已撤销关系不会再被查询到”
- “`IsGuardian` 自动排除了历史撤销关系”
- “REST `active=true` 已经稳定可用”

### 3.4 不要默认 proto/合同里写了就都已可用

`identity.proto` 里当前还定义了：

- `UpdateGuardianRelation`
- `IdentityStream.SubscribeUserEvents`
- `IdentityStream.SubscribeGuardianshipEvents`

但运行时边界是：

- `UpdateGuardianRelation` 当前返回 `Unimplemented`
- 没有发现 `IdentityStream` 的运行时注册

因此 proto 里“存在”不等于今天已经能接。

## 4. 当前鉴权与上下文边界

### 4.1 REST identity 面默认是认证态接口，但存在退化路径

[../../api/rest/identity.v1.yaml](../../api/rest/identity.v1.yaml) 当前有全局：

- `security: - bearerAuth: []`

而 [../../internal/apiserver/interface/uc/restful/router.go](../../internal/apiserver/interface/uc/restful/router.go) 会在 `identity` 路由组上统一挂 `deps.AuthMiddleware`。  
但 [../../internal/apiserver/routers.go](../../internal/apiserver/routers.go) 也明确写了退化逻辑：如果认证模块没初始化成功，就会下发一个 `c.Next()` 的空中间件。

所以更准确的口径是：

- 正常装配下，REST identity 面是 JWT 保护的
- 但仓库层面仍存在“认证模块未起时退化为空中间件”的代码路径

### 4.2 REST 更依赖“当前用户是谁”，gRPC 更依赖“你传了谁”

当前差异很明显：

| 面 | 典型参数来源 |
| ---- | ---- |
| REST `/identity/me*`、`children/register` | 从上下文取当前 `user_id` |
| gRPC `GetUser / ListChildren / IsGuardian / AddGuardian` | 显式从请求体传 `user_id / child_id` |

因此如果你的接入方没有用户态 JWT 上下文，而是一个后台服务，通常更适合先看 gRPC，而不是硬把 REST 当成内部服务接口来用。

## 5. 当前合同漂移与实现缺口

### 5.1 参数名、状态码和返回体有几处已知漂移

当前至少有 4 处应该被明确标出来：

| 位置 | 合同/期望 | 当前实现 |
| ---- | ---- | ---- |
| `GET /identity/guardians` 查询参数 | YAML 用 `user_id / child_id` | binder 用 `userId / childId` |
| `POST /identity/children/register` 状态码 | YAML 写 `201` | handler 走 `h.Success(...)`，实际返回 `200` |
| REST 监护关系响应 | `GuardianshipResponse` 有 `revokedAt` 字段 | 当前转换函数没有填充 |
| REST `active` 过滤 | 合同允许 `active` | `filterGuardianshipResults(...)` 当前直接原样返回 |

对应代码在：

- [../../api/rest/identity.v1.yaml](../../api/rest/identity.v1.yaml)
- [../../internal/apiserver/interface/uc/restful/request/guardianship.go](../../internal/apiserver/interface/uc/restful/request/guardianship.go)
- [../../internal/apiserver/interface/uc/restful/handler/child.go](../../internal/apiserver/interface/uc/restful/handler/child.go)
- [../../internal/apiserver/interface/uc/restful/handler/guardianship.go](../../internal/apiserver/interface/uc/restful/handler/guardianship.go)
- [../../internal/apiserver/interface/uc/restful/response/guardianship.go](../../internal/apiserver/interface/uc/restful/response/guardianship.go)

### 5.2 relation 语义当前并不完全一致

当前 relation 在三层之间存在可证明的偏差：

| 面 | 当前输入 | 当前落地结果 |
| ---- | ---- | ---- |
| REST | `guardian` | 应用层 `parseRelation()` 不识别，最终落成 `other` |
| gRPC | `GRANDPARENT` | 先转成 `"grandparent"`，应用层只识别 `"grandparents"`，最终也会落成 `other` |

这意味着今天不能把“REST / gRPC / domain 的 relation 枚举已经完全对齐”讲成现状。

### 5.3 gRPC 命令里的 `OperatorContext` 目前更多是合同字段，不是已证明闭环

`identity.proto` 里的监护命令请求都带了：

- `OperatorContext`
- `reason`

但当前实现里：

- `AddGuardian` 只消费 `user_id / child_id / relation`
- `RevokeGuardian` 只消费 selector 解析出来的 `user_id / child_id`
- `BatchRevokeGuardians / ImportGuardians` 也是围绕前两条命令包装

所以今天不能把“gRPC 监护命令已经完整消费并持久化 operator/reason 审计字段”讲成已实现。

## 6. 接入前核对清单

如果你准备把业务系统接到 IAM 的身份与监护能力上，建议至少先核对这 7 件事：

1. 你的场景是“当前用户上下文”，还是“服务间显式 ID 查询”。
2. 你要接的是 REST 还是 gRPC，它们在当前并不完全等价。
3. 你是否错误地依赖了 `GET /identity/guardians` 这条尚未接到 router 的 REST 合同。
4. 你是否错误地把 `children/register` 当成了原子事务。
5. 你是否默认撤销关系已经不会出现在查询和判定结果里。
6. 你是否默认 relation 枚举在 REST / gRPC / domain 之间完全一致。
7. 你是否把 proto 里的 `UpdateGuardianRelation / IdentityStream` 当成了当前运行面。
