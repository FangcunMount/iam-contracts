# 身份接入与监护关系边界

本文回答：今天外部系统能怎样接入 `iam-contracts` 的身份、儿童与监护关系能力，REST 与 gRPC 各自覆盖到什么程度，以及 IDP / 缓存治理的真实实现边界是什么。

## 30 秒结论

- REST 当前主入口是 `/api/v1/identity/*`，偏“当前登录用户上下文”。
- gRPC 当前只注册 4 个 identity 服务：`IdentityRead`、`GuardianshipQuery`、`GuardianshipCommand`、`IdentityLifecycle`。
- `POST /api/v1/identity/children/register` 已改成单事务组合用例；只要 guardianship 创建失败，child 会整体回滚。
- `GET /api/v1/identity/guardians` 已接入 router，查询参数固定为 `user_id / child_id`。
- guardianship 默认查询语义已经收口为“只看 active”；历史关系要走显式 `IncludingRevoked` 读路径。
- 对外合同不再保留未实现但可见的 identity stream / relation update / external identity link 占位 RPC。

## 当前暴露面

### REST

| 路径 | 说明 | 当前状态 |
| ---- | ---- | ---- |
| `GET /api/v1/identity/me` | 当前用户资料 | 已实现 |
| `PATCH /api/v1/identity/me` | 更新当前用户资料 | 已实现 |
| `GET /api/v1/identity/me/children` | 当前用户监护的儿童 | 已实现 |
| `POST /api/v1/identity/children/register` | 建档并绑定当前用户监护关系 | 已实现，返回 `201` |
| `GET /api/v1/identity/children/{id}` | 读取儿童档案 | 已实现 |
| `PATCH /api/v1/identity/children/{id}` | 更新儿童档案 | 已实现 |
| `GET /api/v1/identity/children/search` | 搜索相似儿童 | 已实现 |
| `GET /api/v1/identity/guardians` | 按 `user_id / child_id` 查监护关系 | 已实现 |
| `POST /api/v1/identity/guardians/grant` | 显式授予监护关系 | 已实现，返回 `201` |

### gRPC

| Service | 说明 | 当前状态 |
| ---- | ---- | ---- |
| `IdentityRead` | `GetUser / BatchGetUsers / SearchUsers / GetChild / BatchGetChildren` | 已注册 |
| `GuardianshipQuery` | `IsGuardian / ListChildren / ListGuardians` | 已注册 |
| `GuardianshipCommand` | `AddGuardian / RevokeGuardian / BatchRevokeGuardians / ImportGuardians` | 已注册 |
| `IdentityLifecycle` | `CreateUser / UpdateUser / DeactivateUser / BlockUser` | 已注册 |

### 当前明确不开放的 identity 能力

- 事件订阅型 gRPC
- 监护关系类型更新 RPC
- 第三方身份绑定 RPC

这些能力已经从 identity proto、SDK 暴露面、ACL 白名单和 README 中同步移除，不再作为“未来可能支持”的对外合同保留。

## 监护关系语义

### relation 词表

当前 REST、gRPC、领域层统一收口为：

- `self`
- `parent`
- `grandparent`
- `other`

### active / revoked 语义

默认公共查询只返回未撤销关系：

- `FindByUserIDAndChildID`
- `FindByUserID`
- `FindByChildID`
- `IsGuardian`

只有显式历史查询路径才会包含 `revoked_at`：

- `FindByUserIDAndChildIDIncludingRevoked`
- `FindByUserIDIncludingRevoked`
- `FindByChildIDIncludingRevoked`

REST `/identity/guardians` 中：

- 不传 `active` 或 `active=true`：只看 active
- `active=false`：包含历史已撤销关系

## `children/register` 的真实边界

当前链路已经改成单事务：

1. 在一个 `UnitOfWork` 事务里创建 `Child`
2. 直接使用同一组 `TxRepositories` 建立 `Guardianship`
3. 任一步失败都整体回滚

因此当前可以准确表述为：

- “注册儿童并自动绑定当前用户监护关系”是原子闭环
- 不再存在 child 落库成功但 guardianship 失败后留下孤儿 child 的窗口

## REST / gRPC 选择建议

| 场景 | 更推荐的入口 | 原因 |
| ---- | ---- | ---- |
| 当前登录用户查看自己的资料/孩子 | REST | 直接消费 JWT 上下文 |
| 服务间按 ID 读用户或儿童 | gRPC | 显式传 `user_id / child_id` 更自然 |
| 服务间做监护关系判定 | gRPC `IsGuardian` | 返回语义更直接 |
| 当前用户为自己建档并自动授监护 | REST `children/register` | 当前没有对应 gRPC child-create 接口 |

## IDP 实现状态矩阵

| 能力 | 当前状态 | 真实落点 |
| ---- | ---- | ---- |
| `wechat` 认证链路 | 已实现 | `authn` 域内已有微信登录策略，依赖 IDP 应用配置与 token 获取 |
| `wecom` 认证链路 | 已实现 | `authn` 域内已有企业微信登录策略 |
| 租户级应用配置 | 已实现但以微信应用为主 | IDP 模块当前对外暴露的是微信应用配置读写与 gRPC `IDPService` |
| token cache | 已实现 | family 固定为 `idp.wechat_access_token` 与 `idp.wechat_sdk` |
| provider 扩展框架 | 未实现为通用平台 | 当前没有统一的多 provider 注册/路由/合同层 |

## 缓存治理边界

当前 `/debug/cache-governance/*` 明确是 inspect-only：

- 有 `catalog`
- 有 `overview`
- 有 `families/:family`
- 没有 purge / invalidate / warmup / force refresh
