# 授权接入与边界

本文回答：外部系统今天能怎样接入 `iam-contracts` 的授权能力，REST 管理面与在线判定面如何区分，鉴权/租户/操作者上下文边界当前是什么，以及哪些话现在还不能讲成已实现。

## 30 秒结论

- `已实现`：当前可明确对外的授权能力是 `REST /api/v1/authz/*` 管理面，覆盖角色、资源、策略、Assignment 管理和策略版本查询。
- `待补证据`：当前仓库没有发现独立 `authz gRPC` 服务，也没有公开的 `Enforce / Allow` 判定接口；如果业务方要做在线权限判定，不能直接假设 IAM 已提供现成 PDP 暴露面。
- `待补证据`：`authz` 路由在 router 层没有统一挂 JWT 中间件，`authz.v1.yaml` 当前也没有声明 `security`；同时 `tenant_id / user_id` 缺失时会回退到 `default / system`。
- `当前风险`：`RequirePermission` 中间件还是 TODO 放行 stub，`changed_by / granted_by` 在合同里是请求字段、在运行时却改读上下文，`/authz/policies/version` 在新租户无版本记录时存在空值风险。
- 更准确的接法是：把 IAM 当成“授权数据管理面 + 策略版本源”；如果你需要在线权限判定，还要先单独核对判定入口、鉴权方式和运行时保护链。

## 1. 先区分两件事：管理面和判定面

当前授权相关能力，最好先拆成两面看：

| 面 | 当前状态 | 说明 |
| ---- | ---- | ---- |
| 管理面 | `已实现` | 角色、资源、策略、Assignment 的创建、更新、删除、列表和版本查询，走 `REST /api/v1/authz/*` |
| 判定面 | `待补证据` | 当前仓库里没有公开 `allow(user, obj, act)` 这类 REST / gRPC 暴露面，也没有独立 `authz gRPC` |

这意味着当前更适合讲成现状的是：

- 你可以把 IAM 当授权数据的管理中心
- 你可以通过 REST 管理角色、资源、策略和 Assignment
- 你可以读取当前策略版本

不能直接讲成现状的是：

- IAM 已经对外提供统一在线判定 API
- IAM 已经提供可直接消费的 `authz gRPC`
- 业务系统只要接一个中间件就已经完成权限强制执行

## 2. 当前可以怎么接

### 2.1 管理面 REST 已经成形

当前 [../../api/rest/authz.v1.yaml](../../api/rest/authz.v1.yaml) 和 router 一起能证明的能力包括：

| 路径族 | 当前作用 |
| ---- | ---- |
| `/api/v1/authz/roles` | 角色管理 |
| `/api/v1/authz/resources` | 资源管理 |
| `/api/v1/authz/policies` | 策略规则管理 |
| `/api/v1/authz/assignments/*` | 主体与角色分配管理 |
| `/api/v1/authz/policies/version` | 当前策略版本查询 |

如果业务系统接 IAM 的授权管理面，优先读这两处：

1. [../../api/rest/authz.v1.yaml](../../api/rest/authz.v1.yaml)
2. [../../internal/apiserver/interface/authz/restful/router.go](../../internal/apiserver/interface/authz/restful/router.go)

第一处回答“合同怎么写”，第二处回答“运行时真的注册了什么”。

### 2.2 策略版本目前更像“拉取式版本源”

当前公开可证明的版本接口是：

- `GET /api/v1/authz/policies/version`

如果你需要做缓存刷新或配置热更新，今天能直接依赖的是这个拉取式 REST 接口。  
更深的版本传播和 Casbin 双写细节，应回到 [../05-专题分析/02-授权判定链路：角色、策略、资源、Assignment、Casbin.md](../05-专题分析/02-授权判定链路：角色、策略、资源、Assignment、Casbin.md) 看，不要在接口层文档里把内部传播链讲成外部契约。

## 3. 当前不要怎么接

### 3.1 不要把 IAM 当成现成的公共 PDP

当前仓库里没有发现：

- `authz` 独立 gRPC 合同
- `authz` gRPC 服务注册
- 公开的 `Enforce / Allow / Explain` 风格 REST 或 gRPC 接口

因此如果业务系统需要在线权限判定，当前更准确的做法是：

1. 先确认你接的是“授权管理面”还是“在线判定面”
2. 如果要在线判定，先补清真正的判定入口，而不是默认它已经存在

### 3.2 不要默认 authz REST 已经统一受 JWT 保护

当前 `authz` 相关边界是：

- [../../api/rest/authz.v1.yaml](../../api/rest/authz.v1.yaml) 没有声明 `security`
- [../../internal/apiserver/interface/authz/restful/router.go](../../internal/apiserver/interface/authz/restful/router.go) 没有在 `authzGroup` 上挂 JWT 中间件
- [../../internal/apiserver/routers.go](../../internal/apiserver/routers.go) 在注册 `authz` 路由时，也没有像 `user` / `suggest` 一样注入 `AuthRequired`

所以不能直接讲成“authz 管理接口当前都已经统一认证”。

### 3.3 不要把 `RequirePermission` 当成现成强制执行链

[../../internal/pkg/middleware/authn/jwt_middleware.go](../../internal/pkg/middleware/authn/jwt_middleware.go) 当前虽然已经有：

- `RequireRole(...)`
- `RequirePermission(...)`

但两者都还是 TODO 后直接 `c.Next()` 的占位实现。  
这意味着今天还不能把它们写成“当前仓库已经落地的权限判定中间件闭环”。

## 4. 当前鉴权、租户与操作者上下文边界

### 4.1 租户和操作者上下文存在默认回退

当前上下文读取逻辑是：

| 字段 | 当前读取方式 | 缺失时回退 |
| ---- | ---- | ---- |
| `tenant_id` | `core.BaseHandler.GetTenantID(c)` | `default` |
| `user_id` | `core.BaseHandler.GetUserID(c)` 再经 `getUserID(c)` 包装 | `system` |

对应代码在：

- [../../pkg/core/handler.go](../../pkg/core/handler.go)
- [../../internal/apiserver/interface/authz/restful/handler/base.go](../../internal/apiserver/interface/authz/restful/handler/base.go)

这对接入方的含义是：

- 如果调用链没有明确注入 `tenant_id`
- 或者 authz 路由前没有统一认证链把 `user_id` 放进上下文

那么写请求仍可能继续执行，只是租户和操作者会退回默认值。

### 4.2 合同字段与运行时来源存在偏差

当前合同和运行时之间，有两处应明确讲出的差异：

| 能力 | 合同写法 | 运行时真实来源 |
| ---- | ---- | ---- |
| 策略变更 | `changed_by` 是必填请求字段 | handler 实际忽略 `req.ChangedBy`，改读上下文 `user_id` |
| 角色授予 | `granted_by` 是必填请求字段 | handler 实际忽略 `req.GrantedBy`，改读上下文 `user_id` |

对应代码在：

- [../../internal/apiserver/interface/authz/restful/dto/policy.go](../../internal/apiserver/interface/authz/restful/dto/policy.go)
- [../../internal/apiserver/interface/authz/restful/dto/assignment.go](../../internal/apiserver/interface/authz/restful/dto/assignment.go)
- [../../internal/apiserver/interface/authz/restful/handler/policy.go](../../internal/apiserver/interface/authz/restful/handler/policy.go)
- [../../internal/apiserver/interface/authz/restful/handler/assignment.go](../../internal/apiserver/interface/authz/restful/handler/assignment.go)

因此更准确的表达不是“调用方提交什么审计字段，系统就按什么记账”，而是“当前运行时优先使用上下文里的操作者信息”。

## 5. 当前已知风险与漂移

### 5.1 策略版本接口有新租户空值风险

当前 `PolicyQueryService.GetCurrentVersion(...)` 会直接把仓储返回值透出；而 [../../internal/apiserver/infra/mysql/policy/repo.go](../../internal/apiserver/infra/mysql/policy/repo.go) 的 `GetCurrent(...)` 在租户没有版本记录时会返回 `nil, nil`。  
[../../internal/apiserver/interface/authz/restful/handler/policy.go](../../internal/apiserver/interface/authz/restful/handler/policy.go) 的 `GetCurrentVersion(...)` 又直接读取 `version.TenantID / version.Version`，没有先做空值保护。

这意味着：

- 对一个从未产生过策略版本的新租户调用 `/authz/policies/version`
- 当前存在空值风险

### 5.2 接口层不应替代专题层

如果你要解释这些问题：

- Casbin `p / g` 规则怎么编码
- Assignment 与 Casbin 的双写顺序
- 版本递增和版本通知怎么发生

应回到 [../05-专题分析/02-授权判定链路：角色、策略、资源、Assignment、Casbin.md](../05-专题分析/02-授权判定链路：角色、策略、资源、Assignment、Casbin.md)。  
接口层这里只回答“能怎么接、边界在哪”，不重复讲内部机制。

## 6. 接入前核对清单

如果你准备把业务系统接到 IAM 的授权能力上，建议至少先核对这 6 件事：

1. 你接的是授权管理面，还是在线判定面。
2. 你依赖的是 `REST /api/v1/authz/*` 合同，还是某个仓库内实现细节。
3. 调用链谁负责注入 `tenant_id` 和 `user_id`。
4. 你是否错误地假设了 authz router 已统一挂 JWT。
5. 你是否错误地把 `RequirePermission` 当成已经可用的强制执行链。
6. 如果要依赖策略版本，是否已经考虑新租户无版本记录的情况。
