# 授权接入与边界

本文回答：外部系统今天能怎样接入 `iam-contracts` 的授权能力，REST 管理面与在线判定面如何区分，鉴权/租户/操作者上下文边界当前是什么，以及哪些话现在还不能讲成已实现。

## 30 秒结论

- **与 QS 竖切的职责分界**（token、快照、`authz_version`、QS 侧 `CurrentAuthzSnapshot` / capability）：见 [06-IAM-QS竖切边界-Token与授权快照.md](./06-IAM-QS竖切边界-Token与授权快照.md)。
- `已实现`：`REST /api/v1/authz/*` **管理面**（角色、资源、策略、Assignment、版本）与 **PDP**（`POST /authz/check`）；**gRPC** `iam.authz.v1.AuthorizationService/Check`；SDK `client.Authz().Check`。
- `已实现`：`/api/v1/authz/*` 除 `/health` 外在 [router.go](../../internal/apiserver/interface/authz/restful/router.go) 上挂与 user 相同的 `AuthMiddleware`（见 [routers.go](../../internal/apiserver/routers.go)）；OpenAPI 的 `security` 声明仍以合同维护为准。
- `当前风险`：`tenant_id / user_id` 缺失时仍会回退到 `default / system`；`changed_by / granted_by` 合同与 handler 实际取值不一致。
- `中间件`：`RequireRole` / `RequirePermission` 在 JWT 中间件已注入 Casbin 时为真实判定；业务路由需自行 `Use`，且 `casbin == nil` 时会失败。
- 更准确的接法是：IAM 提供 **管理 + 单次判定**；批量、Explain、菜单等按需自建或扩展。

## 1. 先区分两件事：管理面和判定面

当前授权相关能力，最好先拆成两面看：

| 面 | 当前状态 | 说明 |
| ---- | ---- | ---- |
| 管理面 | `已实现` | 角色、资源、策略、Assignment 与版本查询，走 `REST /api/v1/authz/*` |
| 判定面 | `已实现`（单次） | `POST /authz/check`、gRPC `AuthorizationService.Check`、Casbin `Enforce` |

这意味着当前更适合讲成现状的是：

- 你可以把 IAM 当授权数据的管理中心
- 你可以通过 REST 管理角色、资源、策略和 Assignment
- 你可以读取当前策略版本，或通过 REST/gRPC/SDK 做**单次** allow 判定

仍不宜默认成立的是：

- IAM 已提供批量判定、Explain、或与前端菜单强绑定的开箱方案
- 任意 HTTP 路由已自动带上 `RequireRole`（需业务显式挂载中间件）

## 2. 当前可以怎么接

### 2.1 管理面 REST 已经成形

当前 [../../api/rest/authz.v1.yaml](../../api/rest/authz.v1.yaml) 和 router 一起能证明的能力包括：

| 路径族 | 当前作用 |
| ---- | ---- |
| `/api/v1/authz/roles` | 角色管理 |
| `/api/v1/authz/resources` | 资源管理 |
| `/api/v1/authz/policies` | 策略规则管理 |
| `/api/v1/authz/assignments/*` | 主体与角色分配管理 |
| `/api/v1/authz/policies/version` | 当前策略版本查询 |
| `/api/v1/authz/check` | PDP：单次策略判定 |

如果业务系统接 IAM 的授权管理面，优先读这两处：

1. [../../api/rest/authz.v1.yaml](../../api/rest/authz.v1.yaml)
2. [../../internal/apiserver/interface/authz/restful/router.go](../../internal/apiserver/interface/authz/restful/router.go)

第一处回答“合同怎么写”，第二处回答“运行时真的注册了什么”。

### 2.2 策略版本目前更像“拉取式版本源”

当前公开可证明的版本接口是：

- `GET /api/v1/authz/policies/version`

如果你需要做缓存刷新或配置热更新，今天能直接依赖的是这个拉取式 REST 接口。  
更深的版本传播和 Casbin 双写细节，应回到 [../05-专题分析/02-授权判定链路：角色&策略&资源&Assignment&Casbin.md](../05-专题分析/02-授权判定链路：角色&策略&资源&Assignment&Casbin.md) 看，不要在接口层文档里把内部传播链讲成外部契约。

## 3. 当前不要怎么接

### 3.1 克制表述：PDP 能力边界

当前已有 **单次** REST/gRPC 判定与 Casbin `Enforce`，但通常仍不包含：

- 批量 `BatchCheck`、策略 Explain、或与具体业务菜单一一对应的内置模型

业务若需要上述能力，应在契约或业务层扩展，而不是默认 IAM 已提供。

### 3.2 authz REST 与 JWT

运行时上，`/api/v1/authz/*`（`/health` 除外）已挂中央 `AuthMiddleware`。  
合同层若 [authz.v1.yaml](../../api/rest/authz.v1.yaml) 尚未逐路径声明 `security`，接入方仍应以**运行时 router** 与部署配置为准做安全审计。

### 3.3 `RequireRole` / `RequirePermission` 的使用方式

[../../internal/pkg/middleware/authn/jwt_middleware.go](../../internal/pkg/middleware/authn/jwt_middleware.go) 在注入 Casbin 后执行真实判定。  
业务路由需自行 `Use(...)`；未注入 Casbin 时会失败。不能把“注册了 authz 模块”自动等同于“所有业务 API 已按资源做鉴权”。

## 4. 当前鉴权、租户与操作者上下文边界

### 4.1 租户和操作者上下文存在默认回退

当前上下文读取逻辑是：

| 字段 | 当前读取方式 | 缺失时回退 |
| ---- | ---- | ---- |
| `tenant_id` | `core.BaseHandler.GetTenantID(c)` | `default` |
| `user_id` | `core.BaseHandler.GetUserID(c)` 再经 `getUserID(c)` 包装 | `system` |

对应代码在：

- [../../pkg/core/handler.go](../../pkg/core/handler.go)
- [../../internal/apiserver/interface/authz/restful/handler/base.go](../../internal/apiserver/interface/authz/restful/handler/base.go)

这对接入方的含义是：

- 如果调用链没有明确注入 `tenant_id`
- 或者 authz 路由前没有统一认证链把 `user_id` 放进上下文

那么写请求仍可能继续执行，只是租户和操作者会退回默认值。

### 4.2 合同字段与运行时来源存在偏差

当前合同和运行时之间，有两处应明确讲出的差异：

| 能力 | 合同写法 | 运行时真实来源 |
| ---- | ---- | ---- |
| 策略变更 | `changed_by` 是必填请求字段 | handler 实际忽略 `req.ChangedBy`，改读上下文 `user_id` |
| 角色授予 | `granted_by` 是必填请求字段 | handler 实际忽略 `req.GrantedBy`，改读上下文 `user_id` |

对应代码在：

- [../../internal/apiserver/interface/authz/restful/dto/policy.go](../../internal/apiserver/interface/authz/restful/dto/policy.go)
- [../../internal/apiserver/interface/authz/restful/dto/assignment.go](../../internal/apiserver/interface/authz/restful/dto/assignment.go)
- [../../internal/apiserver/interface/authz/restful/handler/policy.go](../../internal/apiserver/interface/authz/restful/handler/policy.go)
- [../../internal/apiserver/interface/authz/restful/handler/assignment.go](../../internal/apiserver/interface/authz/restful/handler/assignment.go)

因此更准确的表达不是“调用方提交什么审计字段，系统就按什么记账”，而是“当前运行时优先使用上下文里的操作者信息”。

## 5. 当前已知风险与漂移

### 5.1 策略版本与新租户

仓储层在租户无版本记录时可能返回 `nil`；[policy handler](../../internal/apiserver/interface/authz/restful/handler/policy.go) 的 `GetCurrentVersion` 已将 `nil` 映射为 HTTP 200 且 `version: 0`。接入方仍可把 `0` 视为「尚无策略变更」的语义，与业务期望对齐。

### 5.2 接口层不应替代专题层

如果你要解释这些问题：

- Casbin `p / g` 规则怎么编码
- Assignment 与 Casbin 的双写顺序
- 版本递增和版本通知怎么发生

应回到 [../05-专题分析/02-授权判定链路：角色&策略&资源&Assignment&Casbin.md](../05-专题分析/02-授权判定链路：角色&策略&资源&Assignment&Casbin.md)。  
接口层这里只回答“能怎么接、边界在哪”，不重复讲内部机制。

## 6. 接入前核对清单

如果你准备把业务系统接到 IAM 的授权能力上，建议至少先核对这 6 件事：

1. 你需要管理面、单次 PDP，还是批量/Explain（后者通常需自建）。
2. 你依赖的是 `REST /api/v1/authz/*` 合同，还是 gRPC/SDK 判定路径。
3. 调用链谁负责注入 `tenant_id` 和 `user_id`（缺失时的 `default`/`system` 是否可接受）。
4. 认证模块未启动时，`AuthMiddleware` 可能降级为放行——部署上是否禁止这种组合。
5. 业务路由是否显式挂了 `RequireRole`/`RequirePermission`，以及进程内是否注入了 Casbin。
6. 策略版本 `0` 与业务「无策略变更」语义是否一致。
