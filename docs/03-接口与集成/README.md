- `api/` 是机器契约层，`docs/03-接口与集成` 是对这些合同和接入方式的解释层。
- 这一组文档回答“怎么接、看哪份合同、验证什么”，不替代 `OpenAPI` 或 `Proto` 本身。
- 当前这一组已经有 REST、gRPC、授权接入边界、身份接入边界，以及面向 `qs` 的接入说明；跨层主链路则由 `05-专题分析` 承载，认证链、授权链和监护关系专题都已经建立。

## 1. `03-接口与集成` 这一组文档今天到底负责什么

接口与集成层只回答“外部系统今天应该怎么接”：

- 看哪份 OpenAPI / Proto / SDK 文档
- 什么时候优先走 REST、gRPC 或 SDK
- 今天哪些合同已经能证明，哪些边界还不能讲过头
- 发生漂移时应该回到哪里核对

它不负责重复维护业务对象建模，也不展开运行时装配和跨层专题长链。

## 3. 这一组文档里哪些接入术语值得统一

先统一这几个词：

- `机器契约层`：指 `api/rest/*.yaml` 与 `api/grpc/**/*.proto`
- `解释层`：指 `docs/03-接口与集成/*` 对契约的导航、边界说明与接入建议
- `接入边界`：指今天能怎么接、不能怎么接、需要回代码核对什么
- `消费面`：指业务方真正会调用的 REST / gRPC / SDK 能力
- `漂移`：指 prose、合同、router、handler 或生成物之间的不一致

## 4. 这一组文档的图表节奏应该怎么把握

- 入口页不堆图，只先说明这组文档回答什么问题。
- REST / gRPC 文优先用“契约层与运行时层的关系图”。
- 授权 / 身份接入文优先用“推荐接法”表和边界图，不重复业务域模型图。
- `QS` 接入文优先用“接入决策流程图”，不复制字段级合同。

换句话说，图只在第一次解释“怎么接、信什么、不要怎么接”时出现。
