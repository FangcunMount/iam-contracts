# 核心概念术语

本文回答：`iam-contracts` 当前最常出现的业务术语、架构术语和契约术语分别是什么意思，它们在代码里大致落在哪，以及哪些旧说法今天已经不该继续沿用。

## 30 秒结论

- 业务域最该先记住的 3 组术语是：
  - `authn`：`Account / Credential / Principal / Token / JWKS`
  - `authz`：`Role / Resource / Assignment / PolicyRule / PolicyVersion`
  - `user(uc)`：`User / Child / Guardianship`
- 代码里的 `uc` 对应的就是用户域，而不是一个额外的独立系统。
- 当前更准确的分层读法是：`interface -> application -> domain -> infra -> container`。
- `docs/` 负责解释这些词怎么在系统里协作，`api/` 负责约束接口字段和 RPC 名称；两者不要混成一层。

## 1. 业务域术语

### 1.1 认证域 `authn`

| 术语 | 当前更准确的含义 |
| ---- | ---- |
| `Account` | 可登录账户，是认证域里的账户锚点，不等于用户档案 |
| `Credential` | 认证材料，例如密码、手机 OTP、微信小程序登录材料 |
| `Principal` | 认证成功后的统一输出，后续由 Token 颁发链消费 |
| `Token` | 当前主要指访问令牌 JWT 与刷新令牌 |
| `JWKS` | 对外发布的公钥集合，供业务方本地验签 |
| `Authenticater` | 按认证场景挑策略并返回认证决策的组件 |

当前最容易讲过头的旧说法：

- 不应把当前实现讲成“以 Session 聚合为中心的完整认证平台”
- 不应把所有预留 Claims 都讲成当前稳定输出

### 1.2 授权域 `authz`

| 术语 | 当前更准确的含义 |
| ---- | ---- |
| `Role` | 角色元数据与 Casbin 角色键的来源，不是内嵌权限树的大聚合 |
| `Resource` | 授权资源目录，维护资源键、动作集合和展示信息 |
| `Assignment` | 谁在某个租户下被赋予哪个角色的关系记录 |
| `PolicyRule` | 显式的 Casbin `p` 规则抽象：`sub / dom / obj / act` |
| `PolicyVersion` | 租户级策略版本，用于同步与刷新链的锚点 |
| `Casbin` | 当前授权规则的执行与持久化承载层之一 |

当前最容易讲过头的旧说法：

- 不应把 `authz` 讲成已覆盖批量判定、Explain、菜单模型等「全家桶」能力（当前以单次 `Check` + Casbin 管理链为主）
- 不应在未核对装配的前提下，把 `RequireRole`/`RequirePermission` 讲成「一定已接通」——需 `AuthzModule` 注入 Casbin 且业务路由显式 `Use` 后才生效

### 1.3 用户域 `user(uc)`

| 术语 | 当前更准确的含义 |
| ---- | ---- |
| `User` | 身份锚点，维护姓名、联系方式、状态等基础资料 |
| `Child` | 儿童档案对象，维护基础身份信息和部分档案字段 |
| `Guardianship` | 用户与儿童之间的关系记录 |
| `Relation` | 当前 `Guardianship` 的关系类型，如 `parent`、`self` |
| `RevokedAt` | 关系是否撤销的当前事实锚点 |

当前最容易讲过头的旧说法：

- 不应把用户域讲成已经有“主/次监护人、邀请码、最多 2 个监护人”的完整关系系统
- 不应把 `children/register` 讲成当前已经是原子闭环

## 2. 架构与代码组织术语

| 术语 | 当前仓库里的读法 |
| ---- | ---- |
| `interface` | REST / gRPC 适配层，负责请求转换、参数校验、协议适配 |
| `application` | 用例编排层，协调领域对象与仓储/外部端口 |
| `domain` | 业务规则与领域模型所在层 |
| `infra` | MySQL、Redis、Casbin、JWT、外部客户端等适配器实现 |
| `container` | 模块装配、依赖注入与启动时 wiring |
| `assembler` | 具体模块的装配入口，例如 `authn`、`user` 模块 |
| `UoW` | 当前用户域应用层里的 unit-of-work 抽象 |

与旧概念图相比，今天更值得坚持的说法是：

- 先看真实目录，再看模式名词
- 模式术语要尽快回链到 `internal/apiserver/` 的真实路径

## 3. 契约与集成术语

| 术语 | 当前更准确的含义 |
| ---- | ---- |
| `OpenAPI` | `api/rest/*.yaml`，REST 机器契约层 |
| `Proto` | `api/grpc/**/*.proto`，gRPC 机器契约层 |
| `Swagger 生成物` | `internal/apiserver/docs/swagger.yaml`，从代码生成并参与比对 |
| `metadata` | gRPC 调用里的鉴权、租户等上下文字段承载方式之一 |
| `SDK` | `pkg/sdk`，官方集成侧复用层，不只是薄 client wrapper |
| `JWKS` | 业务方本地验证 JWT 的公钥发布面 |

## 4. 容易混淆的缩写

| 缩写 | 当前含义 |
| ---- | ---- |
| `IAM` | Identity and Access Management |
| `Authn` | Authentication，认证 |
| `Authz` | Authorization，授权 |
| `UC` | User Center；在代码里主要体现为 `internal/apiserver/domain/uc` 这组目录 |
| `JWT` | JSON Web Token |
| `JWKS` | JSON Web Key Set |
| `DDD` | Domain-Driven Design |
| `CQRS` | Command Query Responsibility Segregation |
