# IAM 系统架构总览

本文回答：`iam-contracts` 当前到底由哪些运行入口、核心分层、契约层和主业务域组成，以及读者从哪里验证这些说法。

## 30 秒结论

- 当前主运行单元是 `iam-apiserver`，入口在 `cmd/apiserver/apiserver.go`；它同时承载 REST、gRPC 和主要业务装配。
- 代码主分层是 `internal/apiserver/{interface,application,domain,infra}`，这是理解六边形架构和 DDD 落地的第一入口。
- 当前主业务域是 `authn`、`authz`、`uc`，另有 `idp`、`suggest` 等补充能力；`suggest` 已补现状版正文，`idp` 的解释层仍相对薄。
- `api/rest/*.yaml` 和 `api/grpc/**/*.proto` 是机器契约层，`internal/apiserver/docs/swagger.yaml` 是生成物；提交前可用 `make api-validate` 检查契约漂移。
- 运行配置和部署事实主要落在 `configs/*.yaml`、`build/docker/`、`scripts/`，不是只看概览文档。

| 关注点 | 当前答案 | 真实落点 |
| ---- | ---- | ---- |
| 运行入口 | 单主进程 `iam-apiserver` | `cmd/apiserver/apiserver.go` |
| 核心代码分层 | interface / application / domain / infra | `internal/apiserver/` |
| 主业务域 | authn / authz / uc | `internal/apiserver/domain/{authn,authz,uc}` |

## 1. 系统定位

### 1.1 是什么

IAM (Identity & Access Management) 是为心理健康测评平台提供统一身份认证与访问控制能力的服务层，当前主要承担：

- **统一认证**：面向 Web、运营后台、小程序等场景提供登录、会话、JWT/JWKS 能力。
- **细粒度授权**：围绕角色、策略、资源与分配关系做权限判定。
- **身份与监护关系**：维护用户、儿童、监护关系等业务数据，并对外暴露 REST / gRPC 能力。

### 1.2 技术栈

| 层次 | 技术选型 | 说明 |
| ---- | ---- | ---- |
| 语言 | Go 1.21+ | 主实现语言 |
| Web 框架 | Gin | REST 接口层 |
| 数据库 | MySQL 8.0 | 关系型存储 |
| 缓存 | Redis 7.0 | 会话、缓存 |
| 授权引擎 | Casbin | 权限决策与策略执行 |
| API | REST + gRPC | 双协议对外能力 |

这个图刻意先回答三件事：

- 当前主运行单元是一个 `iam-apiserver` 进程，而不是多个业务服务拼起来的网格。
- REST、gRPC、工具脚本都围绕同一套 application / domain / infra 组织。
- 读者若要验证边界，应回到 `cmd/apiserver/apiserver.go`、`internal/apiserver/` 和 `api/` 契约层。

### 3.1 真实落点

## 4. 当前架构读法

### 4.1 六边形架构在本仓库里的位置

这套仓库的六边形架构，不应只停留在概念图上。当前更实用的读法是：

1. 从 `interface/` 看有哪些对外输入面。
2. 从 `application/` 看每个用例是怎么编排的。
3. 从 `domain/` 看聚合、实体、值对象和领域服务。
4. 从 `infra/` 看 MySQL、Redis、Casbin、Wechat 等适配器如何落地。

如果需要继续下钻，请读 [04-基础设施与运维/01-六边形架构实践.md](../04-基础设施与运维/01-六边形架构实践.md)。

### 4.2 CQRS 在本仓库中的位置

当前仓库确实把 CQRS 当作重要方法论，但在总览层里更重要的是理解它的**使用位置**，而不是背概念：

| 关注点 | 当前读法 |
| ---- | ---- |
| 写路径 | 通常由 interface -> application -> domain -> infra 完成 |
| 读路径 | 读模型、查询接口、批量读取能力更多体现在 application / infra 的职责划分上 |
| 更细设计 | 继续看 [04-基础设施与运维/02-CQRS模式实践.md](../04-基础设施与运维/02-CQRS模式实践.md) |

## 5. 契约与验证

### 5.1 机器契约入口

| 类型 | 路径 |
| ---- | ---- |
| REST | `api/rest/authn.v1.yaml`、`authz.v1.yaml`、`identity.v1.yaml`、`idp.v1.yaml`、`suggest.v1.yaml` |
| gRPC | `api/grpc/iam/authn/v1/authn.proto`、`identity/v1/identity.proto`、`idp/v1/idp.proto` |
| README | `api/README.md`、`api/rest/README.md`、`api/grpc/README.md` |

### 5.2 校验入口

当前仓库已经有一条真实可跑的契约校验链：

- `make docs-swagger`：从代码生成 `internal/apiserver/docs/swagger.yaml`
- `make docs-reset`：按规则从 swagger 回填 `api/rest` 路径
- `make proto-gen`：生成 protobuf 代码
- `make api-validate`：运行 spectral lint，并比对 swagger definitions / routes 与 `api/rest/*.yaml`

因此，总览文档中的任何接口层结论，都不应脱离这条校验链来表述。
