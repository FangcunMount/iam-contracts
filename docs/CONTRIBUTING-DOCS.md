# iam-contracts 文档写作约定

本文约定 `docs/` 下的现行文档写什么、写到什么粒度、如何与代码和契约对齐。读者入口见 [README.md](./README.md)。

---

## 适用范围

- 本文适用于 `docs/*` 下的现行文档。当前仓库的现行主结构是：
  - `00-概览`、`01-运行时`、`02-业务域`、`03-接口与集成`、`04-基础设施与运维`、`05-专题分析`
- `api/` 下的 OpenAPI、Proto、配套 README，以及 `internal/apiserver/docs/swagger.yaml` 不按本文的 prose 结构要求撰写，但它们属于本文定义的**事实来源**。
- 当前仓库已经建立 `_archive/`，已被新正文替代的旧目录默认进入 `docs/_archive/`，并且**不适用本文结构要求**；迁回现行正文前必须重新核对源码与契约。

迁移期补充约定：

- 新旧目录并存时，以 `docs/README.md` 中标注的阅读路径为准。
- 同一主题一旦已经迁入新目录，旧目录中的同名文档应优先归档到 `_archive/`，避免双份正文长期并存。
- 归档时优先保留旧文档原貌；新的现状版正文只写在新目录体系中，不在 `_archive/` 里继续重写。

## 事实来源与优先级

1. **源码与运行时装配** —— `cmd/`、`internal/apiserver/`、`pkg/`；行为、边界、依赖方向以此为准。  
2. **机器可读契约与配置** —— `api/rest/*.yaml`、`api/grpc/**/*.proto`、`configs/*.yaml`、`build/docker/*.yml`、`build/systemd/`。  
3. **生成物与契约校验链** —— `internal/apiserver/docs/swagger.yaml`、`scripts/check-openapi-contracts.py`、`scripts/check-route-contracts.py`、`make api-validate`。  
4. **`docs/` 现行正文** —— 在 1、2、3 未变更的前提下保持连贯。  
5. **旧文档体系与历史资料** —— 仅作信息源、术语源、图示与迁移参考；迁入正文前须逐条对照当前代码与契约。

补充约定：

- `internal/apiserver/docs/swagger.yaml` 是从代码生成并参与比对的**派生工件**，用于帮助发现代码与 `api/rest/*.yaml` 的漂移；它不是单独的 prose 真值层。
- 旧文档里的事实进入现行正文前，必须重新核对当前代码、OpenAPI、Proto、配置或生成物；无法核对则标注为 `待补证据`，不要写成现状。

## 当前体系的亮点与本轮修订目标

当前文档体系已经有不错的基础，但也有一批需要统一修订的共性问题。本节用于统一后续改文档的方向，不是“自我评价”。

| 类别 | 当前亮点 | 当前不足 | 针对性改法 |
| ---- | -------- | -------- | ---------- |
| **分层与主题** | 已形成 `概览 / 认证 / 授权 / 用户 / 基础设施 / 运维` 的分组，读者能按域进入 | 运行时、接口契约、部署运维的入口仍有些分散，`docs/` 与 `api/` 的边界没有被说透 | 在正文里明确“解释层”和“契约层”的分工；目录页先给阅读路径，再给背景 |
| **事实支撑** | `api/rest`、`api/grpc`、`swagger`、配置、校验脚本都已存在 | prose 文档里对“哪一层更权威”的说明还不够硬 | 持续坚持“源码 / 契约 / 校验脚本优先”，关键结论后 1 屏内应能看到锚点、表、图或契约入口 |
| **讲述能力** | 现有文档已经大量使用图、表、伪代码、术语表，适合建立概念图谱 | 一些长文仍然先铺背景、再给结论；有的示意目录与真实代码路径没有并列出现 | 统一改为“先一针见血，再娓娓道来”；若使用伪代码或示意目录，1 屏内必须补真实代码锚点 |
| **代码对齐** | 文档已经明显尝试对齐六边形架构、DDD、CQRS 与 API 契约 | 个别示意结构仍偏概念化，可能与当前目录演进不同步 | 用“示意图/示意目录”与“真实路径/真实契约”双列出现，避免读者把示意结构误读成真实落点 |
| **维护手段** | 已有 `make api-validate`、`make docs-swagger`、`make docs-reset`、`make proto-gen` 等维护入口 | 还没有一份统一约定说明“改什么必须同步改哪些文档/契约” | 在本文明确维护节奏，把 REST、gRPC、配置、部署、长文重构的同步动作写清楚 |

## 表达习惯

- **先一针见血，再娓娓道来**：文首先回答“这篇文档到底解决什么问题”；每个长小节也先给 1～2 句结论，再展开细节。
- **金字塔结构**：先结论、先边界、先关键认识，再进入背景、原理、演化、细节。
- **结构化重组，不是删信息**：长段正文优先改成表、图、流程步骤、`####` 小节，而不是简单压缩内容。
- **图 + 表 + 描述**：优先用 mermaid、对照表、时序表、流程步骤表达结构；文字负责解释图表没有直接表达出来的部分。
- **图要贴着问题放**：图不是装饰，也不是固定模板。不要机械地在文首堆“金字塔图 + 设计图 + 流程图”；应把总览图放在入口层，把设计图放在对象/职责边界小节，把流程图放在链路/事务/调用顺序小节。
- **示意必须回链真实落点**：如果文中用了“伪代码”“示意目录”“模式草图”，要在 1 屏内给出真实代码路径、真实契约文件或配置入口。
- **一处讲透，其他回链**：同一事实不要在引言、图注、正文、FAQ 里完整重复；正文讲透一次，其余位置只保留摘要与链接。
- **现状 / 证据 / 规划分层**：涉及认证链路、会话、多端登录、权限判定、缓存、幂等、安全机制、部署保证时，必须看得出哪些是已实现，哪些只是建议或规划。
- **少贴大段代码**：默认用“路径 + 类型/函数/配置键名”指路；示例只保留关键片段。
- **安全**：文档里不写真实密钥、Token、密码、证书私钥；配置只写键名、位置或脱敏示例。

## 文首固定套路

除纯索引页外，现行长文默认遵循下面的开头顺序：

1. **`本文回答`**：一句话说明本文解决什么问题。  
2. **`30 秒结论`**：用 3～6 条 bullet 或一张小表，把最该记住的事实先说完。  
3. **主图或主表**：给读者一个总图、主链路表、边界表、对象表中的至少一个。  
4. **详细展开**：再进入模型、链路、接口、配置、边界、注意事项。  
5. **证据与回链**：按需给出代码锚点、契约文件、配置键、验证入口。

补充约定：

- 长文如需目录，目录放在 `30 秒结论` 之后，不能把结论压到下方。
- 目录级 `README.md` 不必机械套用全部小节，但至少要先给**阅读路径**和**本组边界**，不要先写长背景。
- 如果现有文档已经有“核心结论”“关键决策”“设计前置”等同类小节，可沿用，不必强制改名。

## 讲解维度：Why / What / Where / Verify

每个需要讲清边界和实现落点的长文，建议同时覆盖下列四个维度。写作时应默认采用：

`先结论 -> 再用 Why / What / Where / Verify 展开`

| 维度 | 回答的问题 | 典型内容 |
| ---- | ---------- | -------- |
| **Why** | 为什么有这个域/机制/接口、它解决什么问题、与相邻方案怎么分工 | 限界与职责、反模式、取舍、设计动机 |
| **What** | 关键概念、接口、事件、配置、术语分别叫什么 | 聚合根、值对象、REST 路径、RPC 名、配置键、错误语义 |
| **Where** | 真实落点在哪 | `internal/apiserver/...`、`api/rest/*.yaml`、`api/grpc/**/*.proto`、`configs/*.yaml`、`build/docker/*.yml` |
| **Verify** | 读者如何验证自己理解与当前实现一致 | `make api-validate`、`make docs-swagger`、`make proto-gen`、路由比对脚本、配置对照、主链路跟读 |

正文里应使用贴切的中文小标题，不必机械写出 `Why / What / Where / Verify` 英文字样；上表主要用于作者自检。

## 目录分层与推荐结构

### `docs/00-概览/`

负责建立全局心智模型，适合放：

- 系统定位、系统边界、主进程与依赖
- 核心术语表
- 阅读路径与主阅读线
- 代码组织与事实来源

不应在这一组里重复维护领域细节、gRPC/REST 接入说明、SDK 用法或部署脚本细节；这些应回链到对应域文档、`03-接口与集成/`、`05-专题分析/`、`api/` 契约或 `04-基础设施与运维/`。

### `docs/02-业务域/`

当前业务域层已经统一迁入新框架：

- `docs/02-业务域/`：承接 `authn / authz / 用户域（代码目录 `uc`）` 现状版正文
- 旧 `docs/01-认证域/`、`docs/02-授权域/`、`docs/03-用户域/` 已归档到 `docs/_archive/`

单篇长文建议优先覆盖：

- 模块边界：负责什么、不负责什么
- 领域模型：聚合、实体、值对象、不变量
- 应用服务：用例级服务与职责
- 接口入口：REST / gRPC / 其它入口如何接入
- 存储与外部依赖：MySQL、Redis、Casbin、微信等
- 关键链路：登录、权限判定、儿童注册、监护授权等
- 验证入口：相关 OpenAPI、Proto、配置、脚本

如果使用“示意目录”或“设计模式示意图”，应补真实代码锚点，例如 `internal/apiserver/domain/authn/...`、`internal/apiserver/application/authz/...`、`internal/apiserver/interface/uc/...`。

### `docs/04-基础设施与运维/`

负责横切机制与运维交付，不负责重复解释具体业务域流程。适合放：

- 六边形架构、CQRS、依赖方向
- 通用安全机制、缓存策略、事务/一致性策略
- 适配器模式、仓储模式、外部客户端集成方式
- 运行时跨域机制
- 端口、TLS、mTLS、证书要求
- Docker / systemd / Nginx 部署
- 数据迁移、初始化、种子数据
- Makefile 维护动作和日常运维入口

与业务域文档的分工是：基础设施与运维文档讲**方法、共性机制与交付入口**，域文档讲**该域如何具体使用这些机制**。

### `docs/05-专题分析/`

负责跨层主链路与关键设计选择，适合放：

- 认证链、授权判定链、监护关系链这类需要同时回链 interface / application / domain / infra 的专题
- “当前保证 / 当前风险边界 / 后续改造方向”这类不适合散落在单个模块页里的内容
- 与运行时、接口、业务域都有交叉的关键设计取舍

与其他层的分工是：专题文档讲**一条链怎么串起来**，而业务域文档讲**这个模块本身是什么**，接口文档讲**合同怎么消费**，运行时文档讲**服务怎么承载这些能力**。

### `api/`

`api/` 不属于 prose 讲解层，但属于机器契约层：

- `api/rest/*.yaml`：REST/OpenAPI 合同
- `api/grpc/**/*.proto`：gRPC/Proto 合同
- 各自的 README：合同使用说明、兼容性规则、调用约定

凡是接口行为、字段、错误语义、认证方式、分页规则发生变化，必须优先检查 `api/` 层，而不是只改 prose 文档。

## 状态标签

涉及行为、保证、成熟度的表述，默认使用下面三类标签：

- `已实现`
- `待补证据`
- `规划改造`

这些标签尤其适用于：

- 安全保证与密钥轮换
- REST / gRPC 幂等、重试、错误语义
- 多端会话、一致性、缓存策略
- 跨系统集成、批量接口、事件流

如果某结论来自设计意图、伪代码或未来方案，而当前代码/契约还没有完全对应，只能写成 `待补证据` 或 `规划改造`。

## 对旧文档体系与历史资料的处理

- 旧文档体系是**信息源**，不是权威来源。
- 旧文档可以提供术语、图示、读者路径、历史背景和遗漏主题线索。
- 旧文档中的事实进入现行正文前，必须重新核对当前代码、OpenAPI、Proto、配置或生成物。
- 旧结构可以参考，但不必原样保留；如果内容暂时迁不进去，优先放到 `_archive/`，不要强塞进现行正文。

## 维护节奏

- 变更行为、字段、路由、RPC、认证方式、错误语义、配置项的 PR，应在同一 PR 或紧邻 follow-up PR 中同步更新相关文档与契约。
- **REST / Swagger 相关变更**：
  - 变更 handler 注解、路由或模型后，运行 `make docs-swagger`
  - 需要从 swagger 同步 OpenAPI 路径时，运行 `make docs-reset`
  - 提交前运行 `make api-validate`
- **gRPC / Proto 相关变更**：
  - 修改 `api/grpc/**/*.proto` 后，运行 `make proto-gen`
  - 同步更新 `api/grpc/README.md` 与相关概览文档
- **配置 / 运维变更**：
  - 修改 `configs/*.yaml`、`build/docker/*.yml`、`build/systemd/`、证书或部署方式后，同步更新 `docs/04-基础设施与运维/*.md`
- **长文重构**：
  - 优先做“结论前置 + 细节保留”的结构化重构
  - 优先把大段描述改成图、表、步骤或 `####` 小节
  - 优先消除“示意结构与真实代码路径脱节”的问题

当前仓库已有契约校验链，但尚未建立专门的 Markdown 卫生检查。后续如新增 `docs-hygiene`，默认只覆盖现行文档，不覆盖未来的 `_archive/` / `legacy/`。
